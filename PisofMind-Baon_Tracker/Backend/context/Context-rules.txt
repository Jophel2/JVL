# Technical Philosophy

# Prioritize These FIRST

| Priority | Principle              |
| -------- | ---------------------- |
| HIGH     | Readability            |
| HIGH     | Simplicity             |
| HIGH     | Separation of concerns |
| HIGH     | Maintainability        |
| HIGH     | Consistency            |
| MEDIUM   | Scalability            |
| LOW      | Premature optimization |

---

# DO NOT OVERENGINEER

Avoid:

* Microservices
* CQRS
* Event sourcing
* Complex generics
* Overuse of inheritance
* “Smart” code nobody understands

For MVP:

* Simple layered architecture is enough
* Simple CRUD is enough
* Simple service logic is enough

# PisofMind OOP Principles Guide

# 1. Single Responsibility Principle (MOST IMPORTANT)

Each class should have ONE responsibility.

---

# Good Example

## ExpenseService

ONLY handles:

* Creating expenses
* Updating expenses
* Deleting expenses

---

## BudgetService

ONLY handles:

* Budget calculations
* Budget validation

---

# Bad Example

```text id="n7zj5i"
FinanceManager
- login()
- createExpense()
- calculateBudget()
- sendNotification()
- generateReports()
```

Too many responsibilities.

---

# PisofMind Rule

If a class name contains:

* Manager
* Processor
* UtilityGod
* HandlerEverything

…it is probably violating SRP.

---

# 2. Keep Controllers Thin

Controllers should ONLY:

1. Receive requests
2. Validate request structure
3. Call services
4. Return responses

---

# Good Controller

```java id="bjgzvm"
@PostMapping
public ExpenseResponse createExpense(
        @RequestBody ExpenseRequest request
) {
    return expenseService.createExpense(request);
}
```

---

# Bad Controller

```java id="6o5g2d"
@PostMapping
public ExpenseResponse createExpense() {

    // validation
    // budget calculation
    // ranking update
    // database logic
    // email logic
}
```

Business logic does NOT belong here.

---

# 3. Services Own Business Logic

ALL business rules belong inside services.

---

# Good

```text id="j1qlp9"
ExpenseService
- validateExpense()
- calculateRemainingBudget()
- createExpense()
```

---

# DO NOT Put Logic Inside

* Controllers
* React components
* Repositories
* Entities

---

# 4. Repositories ONLY Access Data

Repositories should:

* Save
* Delete
* Query

ONLY.

---

# Good

```java id="zkfyg8"
expenseRepository.findByUserId(userId);
```

---

# Bad

```java id="6h1l2t"
expenseRepository.calculateBudgetAndRank();
```

Repositories should NOT contain business logic.

---

# 5. DTOs are REQUIRED

Never expose entities directly to APIs.

---

# Good Structure

```text id="y7a04o"
ExpenseEntity
ExpenseRequestDTO
ExpenseResponseDTO
```

---

# Why?

DTOs:

* Protect database structure
* Prevent accidental leaks
* Allow API flexibility
* Improve maintainability

---

# 6. Use Clear Naming

# Good Names

```text id="t6vjg4"
ExpenseService
BudgetSummaryResponse
CreateExpenseRequest
```

---

# Bad Names

```text id="tr1u29"
Manager
Handler
Data
ProcessThing
```

---

# PisofMind Naming Rule

A class name should explain:

* WHAT it is
* WHAT it does

without opening the file.

---

# 7. Keep Methods Small

Each method should do ONE thing.

---

# Good

```java id="fjh3xs"
validateBudget()
calculateRemainingBalance()
updateUserBadge()
```

---

# Bad

```java id="w9jlwm"
createExpenseAndCalculateBudgetAndUpdateBadgeAndNotifyUser()
```

Split responsibilities.

---

# 8. Use Composition Over Inheritance

Avoid deep inheritance trees.

---

# BAD

```text id="bqvk8n"
User
 └── Student
      └── SaverStudent
           └── EliteSaverStudent
```

---

# BETTER

```java id="2md0y2"
class User {
    private Badge badge;
}
```

Simple and flexible.

---

# 9. Avoid “God Classes”

If one class controls everything:

* It becomes impossible to maintain
* Impossible to test
* Dangerous to modify

---

# PisofMind Rule

If a class exceeds:

* 300+ lines
* Too many dependencies
* Too many methods

split it.

---

# 10. Organize by Feature, Not by Type Only

---

# Better Structure

```text id="i4k7ae"
features/
 ├── auth
 ├── expenses
 ├── budgets
 └── gamification
```

---

# Instead of Giant Shared Folders

```text id="np4cza"
controllers/
services/
repositories/
```

Feature grouping scales better.

# Entity Design Guide

Entities represent:

* Database tables
* Relationships
* Persistent state

NOT business workflows.

---

# GOOD ENTITY

```java id="jm1wmf"
@Entity
class Expense {
    private BigDecimal amount;
}
```

---

# BAD ENTITY

```java id="jlwmcb"
@Entity
class Expense {

    public void sendEmail() {}
}
```

Entities should not send emails.

# Code Review Checklist

Before merging code, ask:

## Readability

* Is this understandable quickly?

## Responsibility

* Does each class have ONE job?

## Reusability

* Can this be reused?

## Simplicity

* Is there a simpler approach?

## Coupling

* Are dependencies minimized?

## Maintainability

* Will this scale safely?

# Recommended Developer Mindset

> “Build the simplest solution that is clean, understandable, and scalable enough for the next feature.”

---

# Final Engineering Rule

For every class, ask:

1. What is this class responsible for?
2. Does it do too much?
3. Can another developer understand it quickly?
4. Is it easy to test?
5. Is it tightly coupled?
6. Can it scale safely?

If the answer looks bad:

* simplify
* split responsibilities
* reduce complexity
